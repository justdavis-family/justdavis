# Fail Fast and Loud

## Principle

When encountering truly unexpected states (data corruption, schema mismatches between modules),
  fail immediately with clear error messages.
Don't attempt recovery when fundamental assumptions are violated.

## Rationale

- Silent failures or partial recovery mask serious bugs.
- Users need to know when the system is fundamentally broken so they can report issues.
- Fast failures during development prevent bugs from reaching production.
- Clear error messages enable users to report actionable bug reports.

## Examples

**Good (fails fast with clear messages):**
- Use `panic!` in Rust or `fatalError()` in Swift for "impossible" states.
- Validate data schema compatibility between producer and consumer modules.
- Include diagnostic information in crash messages (what was expected, what was found).
- Document why a failure is considered "impossible" so future developers understand.

**Bad (silently recovers or gives unclear errors):**
- Catching unexpected errors and returning default values.
- Generic error messages like "An error occurred" without context.
- Attempting to parse malformed data with fallback behavior instead of rejecting it.
- Continuing execution when internal invariants are violated.

## When NOT to Fail Fast

- **User input errors** - Handle gracefully with validation messages; users make mistakes.
- **Network failures** - Retry with backoff; network issues are transient.
- **Recoverable resource constraints** - Disk space, memory pressure; try to recover gracefully.
- **Expected error conditions** - Use Result/throws for errors that are part of normal operation.

## Relationship to Other Principles

- Complements **comprehensive error modeling** - explicit errors enable fast failure.
- Works with **strong typing** - type mismatches trigger fast failure at compile time.
- Supports **YAGNI** - simple code fails obviously rather than silently limping along.
