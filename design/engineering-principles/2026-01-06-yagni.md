# YAGNI (You Aren't Gonna Need It)

## Principle

Don't solve problems you haven't yet encountered, build features that no one's asked for, or build
  abstractions when a more direct approach would result in fewer total lines of code.

## Rationale

- Premature abstraction adds complexity without proven value.
- Code that doesn't exist can't have bugs, doesn't need maintenance, and doesn't slow down refactoring.
- Building for hypothetical future needs often leads to incorrect abstractions that must be undone
  later.
- Direct, simple implementations are easier to understand and modify when actual requirements emerge.

## Examples

**Good (YAGNI-compliant):**
- Three similar lines of code that do slightly different things → keep them separate until a clear
  pattern emerges.
- Single use case for a feature → implement it directly without configuration options.
- One data format needed → parse it directly without building a pluggable parser framework.

**Bad (YAGNI violations):**
- Building a plugin system when you only have one plugin and no plans for more.
- Adding configuration options for features that have only one known use case.
- Creating helper functions or utilities for operations performed once.
- Designing for "future scalability" without current performance problems.

## When to Break This Rule

- You have concrete evidence that the abstraction will be needed (e.g., a second real use case exists).
- The cost of adding the abstraction later is prohibitively high (e.g., public API design, database
  schema migrations).
- Industry-standard patterns that prevent common bugs (e.g., using an ORM to avoid SQL injection).

## Relationship to Other Principles

- Complements **fail fast and loud** - simple code fails more obviously than over-abstracted code.
- Supports **comprehensive error modeling** - fewer abstractions mean fewer places for errors to hide.
